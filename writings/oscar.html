Oscar Guidelines: Scraper + Criteria Tree Explorer

Oscar Health publishes clinical guidelines as PDFs. Each PDF describes when a treatment or drug is covered, often in two parts: criteria for first-time (initial) authorization and criteria for continuing or renewing care. This project turns those PDFs into decision trees that show only the initial criteria, so you can browse and expand them in a browser.

Code: github.com/jasroopdhingra/oscar

What the project does

  • Finds all guideline PDF links on Oscar's clinical guidelines page (by reading the page's embedded data, not by scraping the HTML).
  • Downloads each PDF. The links on the listing page point to policy pages, not the PDF files, so the app loads each page and reads the real PDF URL from it.
  • Extracts text from the PDF (first 15 pages, up to 30,000 characters). When the PDF has clear section headers, it keeps only the "initial" section and drops continuation or renewal sections.
  • Sends that text to an LLM (Groq, Llama 3.3 70B) and asks it to output a structured decision tree. Every response is checked against a fixed schema. Valid trees are saved; invalid ones are stored with an error message.
  • Provides a small web UI to list policies, filter by whether they have a tree, open one, and expand or collapse the criteria. AND and OR conditions are shown with different labels.

How the pipeline runs

You run three steps in order:

  1. Discover. Reads the guidelines page and saves each PDF link and title into a SQLite database. Duplicates are skipped (same URL = same policy).
  2. Download. For each policy, fetches the policy page, gets the PDF URL, downloads the file to disk, and records success or failure. Waits 1.5 seconds between downloads to avoid overloading the server.
  3. Structure. For policies that have a PDF but no tree yet, extracts text, trims to initial criteria when possible, calls the LLM, validates the result, and saves the tree or the validation error. Waits 3 seconds between LLM calls and processes up to 10 policies per run (configurable).

Re-running a step does not redo work that already succeeded. Discover ignores URLs already in the database. Download and Structure skip policies that already have a file or a valid tree.

Pipeline components

Component What it does
Scraper Loads the guidelines page and parses the embedded JSON (__NEXT_DATA__). Pulls out PDF links from the listed sections and writes them into the policies table.
Downloader For each policy, loads its page, reads the PDF URL from the same JSON, downloads the PDF, and saves a record (path or error). Retries up to 3 times on failure.
Extractor Reads PDF text with pdfplumber (first 15 pages, 30k characters). Looks for "Initial" and "Continuation" section headers and returns only the initial block when both exist.
Structurer Calls Groq with the guideline title and extracted text. The system prompt defines the tree format and says to output only initial criteria. Responses are validated with Pydantic; the tree or an error is stored.

Why "initial" only

Many guidelines split criteria into "initial authorization" (first time) and "continuation" or "renewal." This project focuses on initial criteria so the trees answer: what must be true for a first-time approval?

That is enforced in two ways:

  • Before the LLM: the extractor looks for headers like "Initial Authorization Criteria" and "Continuation of Therapy Criteria." When it finds both, it keeps only the text between them. When it finds only an initial header, it keeps from that header to the end.
  • In the LLM prompt: the system prompt tells the model to output only initial criteria and to omit continuation, renewal, exclusions, and appendices.

If the PDF has no such headers, all extracted text is sent and the prompt alone limits the output. Guidelines that use different wording (for example "First Authorization") are not pre-filtered and rely on the prompt. PDFs longer than 15 pages or 30,000 characters are cut at that limit; if the initial section is after that, it will be missing.

Output format

The tree is JSON. The root has a title, insurance_name ("Oscar Health"), and a single rules object. Each node has rule_id (e.g. 1, 1.1, 1.2.1), rule_text, and, if it has children, operator (AND or OR) and a rules array. Leaf nodes have only rule_id and rule_text.

How it was built

  • Backend: FastAPI, SQLAlchemy, SQLite. Four services (scraper, downloader, extractor, structurer), then a CLI (run_pipeline.py) and API endpoints to trigger each step or fetch the policy list and detail.
  • Frontend: React, Vite, TypeScript, Tailwind. A list of policies with tabs (All, Structured, Unstructured), a detail page, and a recursive tree component with expand/collapse and AND/OR badges.
  • After the app was in place, the full pipeline was run to build trees for at least 10 guidelines, and the README was updated with setup and run instructions.

Getting started

  1. Backend: cd backend, create a virtualenv, pip install -r requirements.txt. Copy .env.example to .env and set GROQ_API_KEY.
  2. Frontend: cd frontend, npm install.
  3. Run the pipeline: cd backend && python run_pipeline.py. To run steps one by one, use run_pipeline.py discover, download, or structure.
  4. Start the API (uvicorn app.main:app --reload --port 8000) and the UI (npm run dev in frontend). Open http://localhost:5173.

Design choices

  • Using the page's embedded JSON (__NEXT_DATA__) instead of scraping the HTML gives one structured source for all links and avoids breakage when the layout changes.
  • Resolving the PDF URL in two steps (listing page → policy page → PDF URL) matches how Oscar's site works: the listing links to pages, and each page embeds the real file URL.
  • Validating every LLM response with Pydantic means invalid output is stored as an error instead of overwriting good data.
  • Making each step idempotent (skip existing work) lets you re-run the pipeline without redoing finished work.
