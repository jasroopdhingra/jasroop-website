Oscar Guidelines: Scraper + Criteria Tree Explorer

This project scrapes Oscar Health's clinical guideline PDFs, extracts text, and turns "initial" medical necessity criteria into structured JSON decision trees you can browse in a React UI. Code: github.com/jasroopdhingra/oscar

The system lets you:

  • Discover all PDF links from Oscar's guidelines page without DOM scraping (parse __NEXT_DATA__ from the Next.js app).
  • Download PDFs to disk with two-step URL resolution. Listing URLs are Next routes. The real file URL comes from each policy page's embedded JSON (Contentful CDN).
  • Extract text with pdfplumber (first 15 pages, 30,000 characters max), then heuristically slice to the "initial" criteria section when headers like "Initial Authorization Criteria" and "Continuation of Therapy Criteria" are present.
  • Structure that text with Groq (Llama 3.3 70B) in JSON mode. Validate every response against a Pydantic CriteriaTree schema. Store either the tree or the validation error.
  • Browse policies and criteria in the UI. Filter by structured vs unstructured, open a policy, expand and collapse nodes. AND and OR are shown as distinct badges.

How it works

The pipeline runs in three phases. Discover upserts policies into SQLite by unique pdf_url (from __NEXT_DATA__). Download fetches each policy page, reads the Contentful PDF URL from the same JSON, streams the file to backend/pdfs/{policy_id}.pdf, and records status. There is a 1.5 second delay between requests. Structure selects policies that have a successful download but no valid structured output yet, runs extract then LLM, validates with Pydantic, and writes to structured_policies. There is a 3 second delay between LLM calls and a cap of 10 policies per run (configurable). Each step is idempotent. Discover uses INSERT OR IGNORE. Download and structure skip rows that already succeeded.

Components

Component Purpose
Scraper Fetch guidelines page. Parse __NEXT_DATA__. Collect links from Contentful modules (Upcoming Policy Changes, Medical Guidelines, Adopted Guidelines). Upsert into policies.
Downloader Per policy: fetch policy page, extract PDF URL from JSON, stream to disk. Tenacity retries (3 attempts). Record Download with stored_location or error.
Extractor pdfplumber text extraction (max 15 pages, 30k chars). Regex for initial vs continuation section headers. Return only the initial slice or full text if no match.
Structurer Groq chat completion with system prompt (schema + "initial only" rules), user prompt (title + filtered text). Uses response_format: json_object. Pydantic CriteriaTree validation. Store result or validation_error.

Schema and initial-only logic

The target JSON has title, insurance_name ("Oscar Health"), and a single root rules object. Each node has rule_id (dot notation: 1, 1.1, 1.2.1), rule_text, and optionally operator (AND or OR) and rules (child array). Leaf nodes have only rule_id and rule_text.

Initial-only is enforced in two places. In the extractor: match patterns like "Initial Authorization Criteria", "Initial Criteria", "Continuation of Therapy Criteria", "Continuation/Renewal Criteria". If both initial and continuation are found, keep only the text between them. If only initial, keep from that header to end. In the structurer: the system prompt states that the model must output only initial authorization/medical necessity criteria and must not include continuation, renewal, exclusions, or appendices. If the document has no such section headers, the full extracted text is sent and the prompt alone defines the scope.

Known limits: guidelines that use non-standard headers (e.g. "First Authorization") are not pre-filtered and depend on the LLM. PDFs longer than 15 pages or 30,000 characters are truncated. If initial criteria fall beyond that, they are missed.

How it was built

Backend first: FastAPI app, SQLAlchemy models (Policy, Download, StructuredPolicy), SQLite. Then scraper, downloader, extractor, structurer. Then run_pipeline.py CLI (discover, download, structure, or all) and API routes (POST /api/pipeline/discover, download, structure, and GET /api/policies, GET /api/policies/{id}). Frontend: React, Vite, TypeScript, Tailwind. Policies list with tabs (All / Structured / Unstructured), policy detail page, recursive CriteriaTree with expand/collapse and AND/OR badges. Vite proxy to backend. Full pipeline run to populate and structure at least 10 guidelines. README updated with setup, run commands, and initial-only logic.

Getting started

  1. Backend: cd backend, python3 -m venv venv, source venv/bin/activate, pip install -r requirements.txt. Copy .env.example to .env and set GROQ_API_KEY.
  2. Frontend: cd frontend, npm install.
  3. Run pipeline: cd backend && python run_pipeline.py (or run_pipeline.py discover, download, structure separately).
  4. Start API: uvicorn app.main:app --reload --port 8000. Start UI: npm run dev in frontend. Open http://localhost:5173.

Decisions

  • __NEXT_DATA__ over DOM scraping: one JSON payload gives all links and metadata. No dependency on CSS or layout.
  • Two-step PDF URL resolution: listing links point at Next routes. The actual PDF URL is in the policy page's __NEXT_DATA__ (Contentful).
  • Pydantic validation on every LLM response: invalid JSON or schema violations are stored in validation_error. No silent overwrite.
  • Idempotency: discover by unique pdf_url. Download and structure skip rows that already have a successful result.
