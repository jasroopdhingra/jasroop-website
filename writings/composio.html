Building a Tool Dependency Graph for Agentic Workflows

I built a pipeline that turned Composio’s Google Super and GitHub tool schemas into a dependency graph: nodes are tools, edges are “tool A can satisfy an input of tool B.” A three-pass resolution (description refs, param matching, optional LLM) inferred dependencies. I produced JSON plus an interactive Cytoscape viz.

In agentic flows, actions often depend on other actions. GMAIL_REPLY_TO_THREAD needs a thread_id; you get it from GMAIL_LIST_THREADS or another tool. The agent must either run something else first or ask the user. Without a map of which tools provide which inputs, that choice is guesswork. I wanted one artifact: a graph (nodes = tools, edges = “source output fills target input”) for Google Super and GitHub, as data plus a viz.

The pipeline fetched tools from the Composio API, extracted params and descriptions, resolved dependencies in three passes, computed “user-provided” inputs (required params no edge satisfies), then wrote dependency_graph.json and generated graph.html. Run it with Bun: bun run src/index.ts. Flags: --skip-fetch, --no-llm, --skip-viz.

I inferred edges from schemas, not from explicit “tool X depends on Y” in the API. Pass 1: if a param or tool description mentioned another tool by name, I added an edge (high confidence). Pass 2: I normalized names and matched output fields to required inputs; same-service preferred, cross-service only for known-safe IDs (spreadsheet, thread, calendar). Medium confidence. Pass 3 (optional): for tools with no or only low-confidence edges, I asked an LLM which tools in the list could provide the data, then filtered self-loops and invalid slugs. Low confidence but often correct. Then I computed userProvided per node—required inputs no edge satisfied—the “ask the user or orchestrator” list.

Execution order required a DAG. I stripped self-loops and two-node mutual cycles. I left deeper cycles for later; with the LLM pass, a full cycle-break before saving stayed on my roadmap.

I wrote dependency_graph.json — nodes (slug, name, service, userProvided) and edges (source, target, param, method, confidence). graph.html is a Cytoscape viz: filter by service, search, click for details, focus 2-hop neighborhood; edges colored by method, nodes with user-provided inputs get a dashed border.

Key Decisions: three passes with explicit confidence so you can tune or disable a pass; userProvided on the node so “what’s still needed from outside” is explicit; same-service preference in param matching to cut false cross-toolkit links.

Next Steps: full cycle removal after LLM so the graph is always a DAG, a topological sort check before save, unit tests for extract/resolve/user-inputs, LLM retries and validation, and an “execution order” or export view in the viz.

You can access the code here: https://github.com/jasroopdhingra/composio-dependency-graph. Run bun run src/index.ts, open graph.html, and explore.
