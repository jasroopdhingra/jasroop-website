Building a Tool Dependency Graph for Agentic Workflows

I built a pipeline that turned Composio tool schemas (Google Super and GitHub) into a dependency graph. Nodes are tools. Edges mean “tool A can supply an input that tool B needs.” The output is a JSON file and an interactive graph you can open in a browser.

In agentic workflows, one action often depends on another. For example, GMAIL_REPLY_TO_THREAD needs a thread_id. You get that from GMAIL_LIST_THREADS or a similar tool. The agent has to either run another tool first or ask the user. Without a map of which tools provide which inputs, that decision is guesswork.

I wanted a single artifact for Google Super and GitHub: a graph (nodes = tools, edges = “this tool’s output fills that tool’s input”) as both data and a visual.

The pipeline did the following:

  • Fetched tools from the Composio API
  • Extracted parameters and descriptions from each tool
  • Resolved dependencies in three passes (described below)
  • Computed “user-provided” inputs: required parameters that no other tool supplies
  • Wrote dependency_graph.json and generated graph.html

To run it: bun run src/index.ts. Useful flags: --skip-fetch (use cached tools), --no-llm (skip the LLM pass), --skip-viz (skip generating the graph).

The API does not say “tool X depends on tool Y.” I inferred edges from the schemas in three passes:

  • Pass 1: If a parameter or tool description mentioned another tool by name, I added an edge. High confidence.
  • Pass 2: I normalized parameter names and matched output fields to required inputs. I preferred same-service matches. Cross-service links I allowed only for known-safe IDs (e.g. spreadsheet, thread, calendar). Medium confidence.
  • Pass 3 (optional): For tools with no or only low-confidence edges, I asked an LLM which tools could provide the data, then removed self-loops and invalid slugs. Low confidence but often correct.

After that I computed userProvided for each node: the list of required inputs that no edge satisfies. That is the “ask the user or the orchestrator” list.

Execution order required a DAG. I stripped self-loops and two-node mutual cycles. I left deeper cycles for later. A full cycle-break step before saving is still on my roadmap.

dependency_graph.json contains nodes (slug, name, service, userProvided) and edges (source, target, param, method, confidence). graph.html is an interactive Cytoscape graph. You can filter by service, search, click a node for details, or focus on a 2-hop neighborhood. Edges are colored by method. Nodes with user-provided inputs have a dashed border.

Decisions I kept. Three passes with explicit confidence so you can tune or turn off a pass. I put userProvided on each node so “what still has to come from outside” is explicit. I used same-service preference in pass 2 to reduce false cross-toolkit links.

Planned next steps. I will add full cycle removal after the LLM pass so the graph is always a DAG. I will add a topological-sort check before saving. I will add unit tests for extract, resolve, and user-inputs. I will add LLM retries and validation. I will add an “execution order” or export view in the viz.
