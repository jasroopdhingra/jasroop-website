Building a Tool Dependency Graph for Agentic Workflows

When an action needs another action first, the agent needs to know. Here’s how we built a graph that tells it.

TL;DR — We built a pipeline that turns Composio’s Google Super and GitHub tool schemas into a dependency graph: nodes are tools, edges are “tool A can satisfy an input of tool B.” Three-pass resolution (description refs → param matching → optional LLM) infers dependencies; we output JSON + an interactive Cytoscape viz. Use it to know what to run before what, or what to ask the user for.

The problem

In agentic flows, actions often depend on other actions. GMAIL_REPLY_TO_THREAD needs a thread_id. You don’t pull that from thin air—you get it from GMAIL_LIST_THREADS (or search, or another tool). The agent has to choose: either run something else first, or ask the user. Without a map of which tools provide which inputs, that choice is guesswork.

We wanted a single artifact: a graph where nodes are tools and edges are “source tool’s output can fill target tool’s input.” Scope: Google Super and GitHub toolkits. Deliverable: the graph as data plus an interactive visualization.

What we built

A small pipeline: fetch tools from the Composio API → extract params and descriptions → resolve dependencies in three passes → compute “user-provided” inputs (required params no edge satisfies) → write dependency_graph.json and generate graph.html.

Run it with Bun: bun run src/index.ts. Flags: --skip-fetch (use cached tools), --no-llm (deterministic only), --skip-viz (no HTML). One command from zero to viz.

How dependency resolution works

We don’t assume the API gives us explicit “tool X depends on tool Y.” We infer edges from schemas: descriptions, parameter names, and (optionally) an LLM that reads tool docs and suggests data providers.

Pass 1 — Description refs

If a parameter or tool description explicitly mentions another tool (e.g. “use the thread ID from GMAIL_LIST_THREADS”), we add an edge and mark it high confidence. Straightforward and reliable.

Pass 2 — Param matching

We normalize names (e.g. thread_id / threadId) and match “this tool’s output field X” to “that tool’s required input X.” Same-service matches are preferred; cross-service we only connect for known-safe IDs (spreadsheet, thread, calendar, etc.) to avoid nonsense links. Result: medium confidence edges.

Pass 3 — LLM (optional)

Tools that still have no edges or only low-confidence ones get a second look. We send the tool’s name, description, and params to a model (Gemini via OpenRouter) and ask: which other tools in this list can provide the data this one needs? The model returns source/target/param; we filter out self-loops and invalid slugs. Those edges are low confidence but often correct—the kind of thing that’s obvious to a reader of docs but tedious to hardcode.

After that we compute userProvided per node: required inputs that no edge satisfies. That’s the “ask the user or the orchestrator” list.

Cycles and DAGs

Execution order only makes sense if the graph is a DAG. We strip self-loops and two-node mutual cycles. Deeper cycles (e.g. A→B→C→A) are left for a follow-up; if you enable the LLM pass, running a full cycle-break pass before saving is on the roadmap.

Outputs

  • dependency_graph.json — Nodes (slug, name, service, userProvided) and edges (source, target, param, method, confidence). Enough to drive an orchestrator or “what should I run first?” logic.
  • graph.html — Cytoscape.js viz: filter by service, search, click a node for details, focus a 2-hop neighborhood. Edges colored by method (description_ref / param_match / llm); nodes with user-provided inputs get a dashed border.

Decisions we’d make again

  • Three passes, explicit confidence. Description refs and param matching are deterministic and explainable; the LLM fills gaps. Tagging each edge with method and confidence makes it easy to tune or disable a pass.
  • userProvided on the node. Knowing “this tool still needs X and Y from outside” is as important as the edges. We compute it once from the final edge set.
  • Same-service preference in param matching. Reduces false cross-toolkit links (e.g. a generic “id” field) while still allowing safe cross-service IDs where it matters.

What’s next

Full cycle removal after the LLM pass so the saved graph is always a DAG; topological sort validation before write; unit tests for extract, resolve, and user-inputs; and a bit more robustness around the LLM (retries, timeouts, schema validation). The viz could gain an “execution order” view from a topological sort, or export a subgraph/CSV for reporting.

The code can be accessed here: https://github.com/jasroopdhingra/composio-dependency-graph. Run bun run src/index.ts, open graph.html, and explore. If you’re wiring agents to Composio tools, this graph is the map your orchestrator didn’t have.
